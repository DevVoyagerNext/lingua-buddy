# Lingua Buddy 开发路线图

## 1. 实施策略

开发顺序遵循：

1. 先完成不依赖 AI 的基础业务。
2. 再接入翻译、纠错和语音等外部能力。
3. 每完成一个模块就补齐历史记录、权限和测试。
4. MVP 稳定后再开发亮点功能。

不要同时展开所有页面。优先完成一条端到端业务链路，再复制成熟模式到其他模块。

## 2. 建议里程碑

以下按 1 人、6-8 周估算。团队开发时可并行前后端任务。

### 阶段 0：需求确认与词典摸底，1-2 天

已完成：

- 已检查 MySQL `ecdict` 的 14 个字段、索引和数据规模
- 已确认共有 770,611 个唯一词条
- 已确认 `id` 主键、`word` 唯一索引以及 `bnc`、`frq` 索引
- 已确认 `exchange`、`collins`、`oxford`、`tag`、`bnc` 和 `frq` 的含义
- 已确认 `pos`、`detail` 和 `audio` 在当前数据库中没有有效数据

仍需完成：

- 确认当前导入数据的具体来源，并保留 ECDICT MIT License 和署名
- 确认 AI、ASR 供应商
- 确认部署环境
- 使用 10-20 个典型单词验证词典数据质量
- 固化 MVP 范围

完成标准：

- GORM 精确查询和前缀联想可运行，并确认查询能够使用 `word` 索引
- 应用按只读方式访问 `ecdict`
- 外部服务密钥可在测试环境使用

### 阶段 1：工程骨架与用户系统，第 1 周

后端：

- 初始化 Go Module 和 Gin
- 配置、日志、GORM 数据库连接和统一错误响应

- 注册、登录、单 JWT 认证和退出

- 布局、路由、请求封装和 Pinia
- 登录、注册、首次设置
- 路由守卫和统一错误提示

测试：

- 认证 API 集成测试
- 用户数据访问测试

阶段验收：

- 新用户可注册并进入学习首页
- 刷新页面后登录状态可恢复

### 阶段 2：词典、单词笔记、收藏与复习，第 2 周

后端：

- 生词收藏、列表和删除
- 四阶段渐进学习策略和到期训练查询
- `training_answer_records` 答题历史和提交幂等
- 单词错误累计、阶段降级和历史错词查询
- 今日单词学习接口

前端：

- 查词页和词条详情
- 收藏状态交互
- 生词本页
- 单词笔记功能
- 收藏句子页
- 渐进式单词学习页：英文选中文、中文选英文、中文默写
- 首页今日到期单词入口

测试：

- 重复收藏和重复计划词条约束
- 固定队列、每日新增和活跃窗口计算
- 退出后按计划项和用户单词继续学习
- 阶段晋级、降级和复习时间计算
- 干扰项去重、评分、降级和不足候选错误

阶段验收：

- 完成“查词 -> 收藏 -> 英文选中文 -> 中文选英文 -> 中文默写 -> 已掌握”的可演示闭环

### 阶段 3：翻译、语法分析与纠错，第 3 周

后端：

- AI Provider 抽象
- 翻译提示词和结构化输出
- 语法分析提示词和结构化输出
- 语法纠错和润色
- 统一历史表
- 超时、重试和限流

前端：

- 智能翻译页
- 语气选择、结果解释和备选表达
- 语法分析、纠错与润色页
- 错误列表可视化

测试：

- 使用 Mock Provider 测试业务逻辑
- 无效 JSON、超时、限流和空结果
- 历史数据用户隔离

阶段验收：

- AI 供应商替换不会影响 handler 和业务数据结构

### 阶段 4：语音识别与统一历史，第 4 周

后端：

- 音频上传和安全校验

- 同步语音识别
- 识别文本修订
- 识别后翻译
- 原始音频保存与删除

前端：

- 浏览器录音和文件上传
- 识别加载状态
- 文本修改与再次翻译
- 统一历史中心

测试：

- 不同音频格式

阶段验收：

- 完成“录音 -> 识别 -> 修订 -> 翻译 -> 历史查看”流程

### 阶段 5：MVP 收尾，第 5 周

- 空状态、加载状态、错误状态
- 移动端基础适配
- OpenAPI 文档
- 数据库索引检查
- 安全检查和接口限流
- 关键 E2E 测试
- Docker Compose 和部署说明
- 演示数据和演示脚本

MVP 发布标准：

- 产品需求文档中的 MVP 验收清单全部通过
- 无阻断级缺陷
- 核心流程可连续演示
- 外部服务失败时基础学习数据不丢失

### 阶段 6：项目亮点，第 6-8 周

按以下顺序实现：

1. 外刊阅读
2. AI 情景对话
3. 作文批改
4. 翻译和作文专项训练、AI 评价与翻译错题

这样安排的原因是：渐进式单词学习和单词答题记录已经在第 2 周完成；第二阶段只需让翻译、作文训练复用现有 AI Provider 和通用 `training_answer_records`。

## 4. 最小演示剧本

最终答辩建议用一个用户故事串联功能，而不是逐页点击：

1. 用户登录并查看今日任务。
2. 查询 `accomplish`，查看音标、释义和例句。
3. 将单词加入生词本，标记为“不认识”。
4. 输入一句错误英文，展示 AI 纠错及中文解释。
5. 录制一句中文，展示 ASR 文本，再翻译为自然英文。
6. 在历史中心查看刚才的查词、纠错和语音记录。
7. 进入复习页完成一次单词复习。
8. 展示学习首页数据变化。
9. 时间允许时，进入 AI 面试场景完成两轮对话。

这个演示同时体现 Go 后端业务、数据库、AI 接入、语音处理和学习闭环。

## 5. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| `pos`、`detail`、`audio` 无有效数据 | 词性、例句和音频无法直接展示 | 从释义前缀展示词性；例句和音频作为独立增强 |
| AI 输出格式不稳定 | 页面报错或数据污染 | 使用结构化输出、服务端校验和有限重试 |
| AI 成本过高 | 无法持续演示 | 限流、字符上限、小模型优先、Mock Provider |
| ASR 处理时间长 | 用户以为系统卡死 | 异步任务、进度状态、可重试 |
| 音频格式兼容问题 | 上传失败 | 前端统一录音格式；后端必要时转码 |
| 功能范围过大 | 无法按期完成 | 锁定 P0；第二阶段按顺序开发 |
| 历史数据泄露 | 严重安全问题 | 所有查询强制带当前用户 ID；增加隔离测试 |
| 外部服务不可用 | 演示中断 | Mock 模式、超时降级、预备演示数据 |

## 6. 开发纪律

- 每个功能先定义请求、响应和错误，再实现前后端。
- 专项训练必须区分“获取题目”和“提交答案”：获取或跳过不入库，提交成功必须入库。
- AI 训练先保存用户答案，再调用外部 Provider；外部调用失败不能丢失答案。
- 业务表结构通过 GORM 模型和独立迁移命令管理。
- Repository 统一使用 GORM，不使用 `Raw`、`Exec` 或手写完整 SQL。
- AI 和 ASR 在开发测试中支持 Mock Provider。
- 每个模块完成时同步更新 OpenAPI。
- 新增或更换外部资料源、AI、ASR、TTS 或对象存储时，同步更新 `docs/04-data-sources-and-external-services.md`。
- 修改单词阶段、题型或干扰项算法时，同步更新 `docs/05-progressive-word-learning.md`。
- 修改词汇计划、批量激活或断点续学时，同步更新 `docs/06-word-learning-plan-and-resume.md`。
- 未在数据来源清单登记的第三方接口不得直接进入生产代码。
- 合并前至少运行后端测试、前端类型检查和构建。
- 不把密钥、真实用户音频和生产数据提交到 Git。
- 当前版本只实现面向学习用户的功能。

## 7. 首批开发任务

开始编码时建议按以下顺序建立任务：

1. 根据实测结构建立完整的 `ECDICTEntry` GORM 模型。
2. 将现有命令行查词原型重构为 GORM，并迁入 `backend` 词典模块。
3. 初始化 Gin API 和 Vue 前端工程。
4. 建立配置、日志、GORM 连接、错误响应和健康检查。
5. 建立用户 GORM 模型、迁移命令和认证流程。
6. 编写 `ecdict` GORM Repository 和前缀联想。
7. 完成查词 API 和页面。
8. 建立 `word_learning_plans`、`word_learning_plan_items` 和 `user_words`。
9. 实现 CET-4 固定队列、每日新增和活跃窗口。
10. 实现 recognition -> discrimination -> spelling -> mastered 阶段策略。
11. 实现独立 `worddistractor` 模块和两种四选一题型。
12. 建立 `training_answer_records`，实现断点续学和历史错词查询。
13. 完成第一条端到端学习闭环。

在第 13 项完成前，不建议开始作文批改。
